# Runtime Isolation — How User Code Actually Runs

> **What this is:** the deep dive on the physical runtimes Lantern uses to execute agent code, and which one to pick for which workload. Read this if you're touching `services/runtime-manager` or any of the `runtimes/` subdirs.
>
> **Why it matters:** the wrong isolation choice is the difference between a serverless platform and a security incident.

---

## The five isolation classes

Every agent declares an `isolation.class` in its `agent.yaml`. The runtime manager translates this into a physical backend.

| Class | Backend | Cold start (target) | Warm start (target) | When to use |
|---|---|---|---|---|
| `trusted` | K8s Job pod | ~1.5s | ~100ms | Internal Lantern steps; first-party agents owned by the tenant; no untrusted code |
| `standard` | Firecracker microVM via firecracker-containerd | ~1.5s cold, **~150ms warm (snapshot)**, **~28ms ideal** | 150ms | **Default.** Tenant-owned agents that may load packages or call external APIs |
| `untrusted` | Firecracker + seccomp deny-default + egress allowlist + readonly rootfs | ~1.5s | ~150ms | Code generated by an LLM, user-uploaded scripts, anything not signed |
| `hostile` | Kata Containers via K8s RuntimeClass | ~2s | ~300ms | Attacker-controlled inputs that exercise kernel surfaces; CTF-style workloads |
| `wasm` | Wasmtime in-process inside a trusted host | ~5ms | ~5ms | Pure-function tools with no I/O, deterministic transformations |
| `devcontainer` | Long-lived pod with attached PVC | ~3s (one-time) | n/a | Agents that maintain persistent workspaces (think: an agent that owns a Git repo and works on it for days) |

## Why these specific backends

### Firecracker is the default for ephemeral untrusted execution

Verified production-grade in 2026 ([Firecracker snapshotting docs](https://github.com/firecracker-microvm/firecracker/blob/main/docs/snapshotting/snapshot-support.md), [28ms boot blog post](https://dev.to/adwitiya/how-i-built-sandboxes-that-boot-in-28ms-using-firecracker-snapshots-i0k)). Why:

- **Hardware-level isolation.** Each microVM has its own kernel. A guest exploit cannot reach the host kernel without also breaking out of KVM — a much higher bar than escaping a Linux namespace.
- **MAP_PRIVATE snapshot restore.** The runtime manager memory-maps the snapshot file into the VM's address space; pages are loaded on demand. There is no kernel boot, no init, no agent startup — execution resumes from exactly where the snapshot was taken. **AWS Lambda SnapStart uses this exact mechanism.**
- **Tiny resource overhead.** ~5 MB of memory per microVM beyond the guest kernel. We can pack hundreds per node.
- **Mature operations** via firecracker-containerd, the AWS-maintained shim that lets containerd manage Firecracker the same way it manages OCI containers. This is what lets us use K8s for scheduling without K8s knowing or caring about microVMs.

E2B's production deployment uses Firecracker for the same reason and achieves ~150ms boot in production at scale ([E2B](https://e2b.dev/)). We target the same.

### Kata Containers for the truly hostile workloads

Kata is heavier than Firecracker (it boots a fuller guest with more devices) but uses the standard K8s `RuntimeClass` mechanism, so it's a one-line addition per pod and doesn't require firecracker-containerd in the data path. Kata is what we run when the threat model says "this code is actively trying to escape." See [State of MicroVM Isolation 2026](https://emirb.github.io/blog/microvm-2026/).

### Wasmtime for pure-function steps

Wasm is dramatically faster (microseconds, not milliseconds) but trades off everything else: no syscalls, no real filesystem, no network beyond what the host explicitly grants via WASI. For compute-only transformations (parsing, validation, scoring) it's perfect. For anything that touches I/O, you want Firecracker.

### K8s Job for trusted internal code

For first-party agents we wrote ourselves and signed, we don't need microVM isolation; the Linux user namespace + seccomp + the K8s NetworkPolicy is enough. K8s Job gives us the cheapest cold start path and the simplest debugging story.

### devcontainer for long-lived workspaces

Some agents are not "spin up, do a thing, tear down." A coding agent that's been working on your repo for three days needs to keep its `node_modules`, its cached compilation, its open editor state. We back these with a long-lived pod and an attached PVC. The runtime manager pools and reaps idle workspaces.

Each backend implements:

```rust
#[async_trait]
pub trait RuntimeBackend: Send + Sync {
    fn class(&self) -> IsolationClass;

    async fn schedule(&self, plan: &SchedulePlan) -> Result<RuntimeHandle>;

    async fn cancel(&self, handle: &RuntimeHandle) -> Result<()>;

    async fn stream(&self, handle: &RuntimeHandle) -> Result<EventStream>;

    async fn snapshot(&self, handle: &RuntimeHandle) -> Result<SnapshotRef>;

    async fn restore(&self, snapshot: &SnapshotRef, plan: &SchedulePlan)
        -> Result<RuntimeHandle>;
}
```

Adding a new backend (e.g. Cloudflare Containers, gVisor) is a single `impl RuntimeBackend`.

## Snapshot / restore for fast cold starts

This is where Lantern wins on cold start. The model:

1. **The first time** an agent version runs, the runtime manager:
   - Cold-boots a Firecracker microVM
   - Loads the agent bundle
   - Runs the agent's `init` hook (the per-version warmup: import deps, load config, open DB pools)
   - Pauses the VM at the moment the agent is ready to receive its first input
   - Snapshots the VM state to S3 (`s3://lantern-snapshots-prod/runtimes/firecracker/<digest>/snapshot.fcs`)
2. **Every subsequent run** of that version:
   - Looks up the snapshot by `(agent_version_digest, isolation_class)`
   - Restores the VM via `MAP_PRIVATE` mmap of the snapshot file (no copy, no boot)
   - Injects the run-specific input + secrets via the secrets channel
   - Resumes execution

**Result:** the user's `run` function receives its first input ~150ms after the schedule call, with all imports already done and DB pools already open. For warm-pool hits (the snapshot is already mmap'd into a pre-spawned VM that's just waiting for an input), we target ~28ms.

The runtime manager keeps a per-(class, digest) **warm pool** of restored-but-idle VMs. Pool size is dynamic based on the recent rate of runs for that version (mini PID controller).

Snapshots are invalidated when the agent version changes (different digest → different cache key). They're also evicted from S3 by lifecycle policy if unused for N days.

### The init hook

```ts
// In agent code
import { agent } from "@lantern/sdk";

export default agent({
  name: "my-agent",
  async init() {
    // Runs ONCE per version, before the snapshot is taken.
    // Use this to import heavy deps, warm caches, open pools.
    await loadEmbeddings();
    await connectToVectorStore();
  },

  async run({ input, ctx }) {
    // Runs every time. Snapshot was taken at the start of this function.
    return doTheWork(input);
  },
});
```

Anything that lives in `init()` is paid once per version. Anything in `run()` is paid per run but starts from the post-init state.

## Untrusted class hardening

For `untrusted`, on top of Firecracker's isolation we add:

1. **Read-only rootfs.** The bundle is mounted read-only. Writes go to a tmpfs or a per-run scratch volume.
2. **seccomp deny-default profile.** Only the syscalls the runtime needs are allowed. Profile is in `runtimes/firecracker/seccomp/untrusted.json`. We deny `ptrace`, `unshare(CLONE_NEWUSER)`, `bpf`, `kexec_load`, `process_vm_*`, etc.
3. **Egress allowlist enforced two ways:**
   - At the network namespace via NetworkPolicy + a per-tenant CNI ACL
   - In the agent SDK's HTTP client via a host whitelist (defense in depth)
   - Default allowlist: nothing. Agents must declare `egress: [...]` in `agent.yaml`. Built-in `lantern.web` tool has its own approved egress list.
4. **Memory + CPU caps** from `agent.yaml.limits` translated to cgroups + Firecracker MMIO config.
5. **Filesystem quotas** on the scratch volume (default 1 GiB).
6. **Time bomb.** Hard timeout from `agent.yaml.limits.timeout`; runtime manager kills the VM unconditionally at that mark.
7. **Secrets injection via tmpfs** at exec time, never written to disk that survives the run.
8. **No /dev/kvm passthrough.** No device passthrough at all.

## Hostile class hardening (Kata)

Everything `untrusted` has, plus:

1. **Dedicated nodes.** Hostile workloads run on a separate node pool (`node-pool: hostile`). No co-tenancy with anything else.
2. **K8s RuntimeClass `kata-clh`** (Kata + Cloud Hypervisor) with QEMU isolated by default.
3. **No host network access at all** for the pod's network namespace; all traffic goes through a per-pod proxy that only forwards explicitly-allowlisted destinations.
4. **eBPF monitoring** on the host for syscall anomalies; suspicious patterns trigger an immediate kill.
5. **Mandatory disposable scratch.** No PVCs allowed.

## GPU workloads

For agents that need a GPU (training, fine-tuning, large image/audio model inference) the runtime manager has a separate scheduler that places them on GPU nodes:

- **NVIDIA device plugin** for K8s
- **MPS (Multi-Process Service)** for fractional GPU sharing on inference workloads
- **MIG (Multi-Instance GPU)** on A100/H100 nodes for hard GPU partitioning when isolation matters
- **GPU memory snapshotting** (Modal-style) on hosts that support it — alpha for now, behind a feature flag

GPU scheduling respects:
- Per-tenant GPU quota
- Spot vs on-demand preference (`limits.gpu.spot: true`)
- Region affinity (where the data is)

GPU isolation: GPU workloads run as `trusted` or `standard` because Firecracker doesn't pass GPUs through. For untrusted GPU code we require the user to wrap it in a proxy that validates inputs at the boundary.

## Computer use sandbox

A separate runtime profile for agents that drive a real desktop (browser automation, RPA, computer-use agents):

- Backed by a long-lived pod running Xvfb + a window manager + a Chromium instance
- Stream of compressed frames (VP9 ~10 FPS) goes to the surface-gateway for live screen-share
- Inputs (mouse, keyboard) come back from the surface-gateway via the same stream
- Egress allowlist still enforced; the browser is locked to allowlisted domains
- Per-run cookie / profile isolation; nothing persists across runs unless explicitly saved to memory

This is what powers the "agent that does things in your browser while you watch on your phone" pattern.

## What about cold-start latency targets

| Phase | Target | Achieved how |
|---|---|---|
| Warm-pool hit (snapshot pre-mapped, VM idle, just inject input) | **≤ 30 ms** | Pool of pre-restored VMs per (class, digest) |
| Snapshot restore (no warm pool, but snapshot exists) | **≤ 200 ms** | MAP_PRIVATE mmap of S3-cached snapshot file |
| Cold cold (first time this version runs anywhere) | **≤ 1.5 s** | Pre-pulled base images, lazy bundle load |
| GPU model load (with snapshot) | **≤ 2 s** | GPU memory snapshotting |
| GPU model load (without) | **≤ 30 s** | Best effort; user pays once per version |

These are p50 targets for the spike. Production p99 targets are stricter and tracked in the SLO doc.

## What's intentionally NOT in this design

- **No bring-your-own-K8s for the data plane in personal mode.** Personal users always run on Lantern-managed infrastructure. Enterprise can BYO clusters.
- **No nested virtualization.** Firecracker requires KVM; we run on bare-metal or KVM-enabled hosts only. Cloud providers' nested-virt instances are not supported.
- **No Wasm for general agent code.** Wasm is for tools, not for whole agents — the WASI surface is too small.
- **No "magic" snapshot restoration of agents that hold network connections.** Snapshots happen at the start of `run()`, not arbitrary points. If your agent holds a long-lived connection across `init` → `run`, it must reconnect inside `run`.
